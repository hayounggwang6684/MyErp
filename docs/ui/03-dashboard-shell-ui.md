## 1. 목적

이 문서는 로그인 성공 후 Electron 클라이언트에서 보이는 첫 ERP 대시보드의 공통 레이아웃과 8개 상위 탭 구조를 정의한다.

기본 시각 기준은 로그인 화면과 동일한 `03 미니멀 코퍼레이트` 계열이다.

## 2. 진입 경로

- Electron 클라이언트 내부 대시보드
- `/admin/dashboard`

일반 사용자 브라우저 경로 `/dashboard` 는 현재 업무 UI를 제공하지 않고 클라이언트 안내 페이지로 대체된다.

## 3. 전체 레이아웃

```text
+------------------------------------------------------------------+
| TopBar: 화면 제목 | 사용자 | 버전 | 업데이트 확인 | 로그아웃      |
+------------------------------------------------------------------+
| TabBar: 고객관리 | 주문관리 | 공사관리 | 자산관리 | 부품관리 | 청구관리 | 직원관리 | 설정 |
+------------------------------------------------------------------+
| SectionHeader + Quick Actions                                    |
+------------------------------------------------------------------+
| Summary Cards / Recent Table / Personal Settings                 |
+------------------------------------------------------------------+
```

## 4. 상단 바

구성:

- 로고
- 현재 화면 제목
- 사용자 이름
- 역할 배지
- 알림 영역
- 로그아웃 버튼

원칙:

- 상단 바는 얇고 정돈된 형태
- 불필요한 큰 배너 금지
- 흰색 또는 매우 옅은 배경 위 얇은 선과 브랜드 블루 포인트만 사용

## 5. 상위 탭

고정 탭:

- 고객관리
- 주문관리
- 공사관리
- 자산관리
- 부품관리
- 청구관리
- 설정

조건부 탭:

- 직원관리

규칙:

- 기본 진입 탭은 `주문관리`
- `직원관리`는 로그인 사용자의 `department`가 `관리부`이거나 `STAFF_VIEW` 권한이 있을 때 보인다
- `고객관리`, `주문관리`, `공사관리`, `자산관리`, `부품관리`, `청구관리`도 각각 역할 기반으로 노출한다
- `설정`은 개인화 전용이며 관리자 보안 기능은 포함하지 않는다
- 탭은 상단 바 바로 아래에 배치하고, 활성 탭은 브랜드 블루 배경으로 표시한다

## 6. 탭별 본문 블록

### 6.1 고객관리

- 본문 구조: 상단 검색바, 왼쪽 고객 목록, 오른쪽 고객 상세 패널을 기준으로 한다.
- 고객 목록은 업체명과 대표/담당자 기준으로 표시하며, 행 선택 시 오른쪽 상세 패널을 갱신한다.
- 상세 패널은 `기본 / 조직 / 장비` 하위 탭을 사용한다.
- 기본 탭은 회사 기본정보와 메모를 같은 조회 레이아웃에서 보여준다.
- 신규 등록은 조회 화면과 동일한 필드 배치를 사용하되 값만 비어 있고 입력 가능한 상태로 전환한다. 신규 모드에서는 `새로고침` 버튼을 `등록 취소`, `신규` 버튼을 `저장`으로 바꾼다.
- 등록 완료된 회사 정보는 `우클릭 수정` 패턴을 사용한다. 조회 상태의 특정 필드를 우클릭하면 해당 필드만 인라인 수정 가능 상태로 전환하고 저장/취소 후 다시 읽기 전용으로 돌아간다.
- 장비 탭은 왼쪽 선박 목록과 오른쪽 장비 목록/장비 신규 추가 영역으로 구성한다.
- 선박 목록과 장비 목록은 각각 독립 스크롤 영역을 가진다. 항목이 많아져도 전체 앱 스크롤을 만들지 않고 해당 목록 내부에서만 스크롤한다.
- 선박과 장비 항목은 개별 삭제 액션을 제공한다. 삭제 전 확인을 거치고 삭제 후 선택 상태와 빈 상태 문구를 갱신한다.
- 고객관리 내부 패널은 상단 검색 영역 아래 남은 세로 공간을 항상 채우며, 내용이 많을 때는 패널 내부에서만 스크롤한다.

### 6.2 주문관리

- 본문 구조: 주문관리 내부 페이지 탭, 기간 조회 검색바, 왼쪽 주문 목록 그리드, 오른쪽 주문 내역/공사 상세 패널을 기준으로 한다.
- 내부 페이지 탭은 상위 업무 탭과 구분되는 주문관리 전용 하위 탭이다. 기본 탭은 `주문등록`이며 향후 `견적 / 공사 / 납품` 같은 업무 페이지를 열어 본 페이지로 유지할 수 있게 한다.
- 기간 조회 검색바는 시작일, 종료일, 조회, 초기화만 제공한다. 선사/선박/엔진 같은 세부 필터는 초기 화면에 넣지 않는다.
- 주문 목록 컬럼: 주문일, 거래처, 선박, 구분, 수주, 주문명, 상태, 주문ID, 공사ID
- 오른쪽 상세는 `주문 내역 / 공사` 하위 탭을 가진다.
- 주문 내역 필드: 매출처, 선사, 선박, 엔진, 주문구분, 주문명, 주문상태, 설명, 주문담당자, 부서구분, 수주 체크, 비고, 등록일시, 등록자, 수정일시, 수정자
- 문서 목록 컬럼: 문서종류, 문서명, 문서ID
- 공사 탭 필드: 관리번호, 공사구분, 공사명, 진행상태, 착수일, 완료예정일, 담당부서, 담당자
- 상태 흐름: 요청 접수, 견적 진행, 확정 대기, 공사 등록, 납품 등록, 취소
- 관리번호 규칙: 확정 체크 후 `SH-YYYY-NNN-T` 또는 `SH-YYYY-NNN-S` 형식으로 표시한다. `T`는 공사, `S`는 일반 판매/납품이다.
- 관리번호는 한 번 발급되면 재정렬하지 않는다. 삭제/취소/병합으로 중간 번호가 비어도 결번으로 유지한다.
- 신규 관리번호는 해당 연도에 이미 사용된 최대 순번 다음 번호를 발급하며, 중간 빈 번호는 재사용하지 않는다.
- 수리 공사에 쓰이는 부품은 `T` 건 안에 포함하고, 별도 납품/판매 건은 `S` 건으로 분리 등록할 수 있게 한다.
- 화면은 기존 ERP 주문등록처럼 고밀도 업무 그리드와 상세 입력 폼 중심으로 구성하며, 리본 도구창은 구현하지 않는다.
- 초기 구현은 프론트 시연 상태를 허용하되, 데이터 구조는 향후 주문 API 연결을 전제로 둔다.

### 6.3 공사관리

- 견적 수주 이후 공사 문서, 체크리스트, 진행 상황, 레포트, 외부 의뢰, 준공 문서를 관리한다.
- 화면 구조는 공사뷰/문서뷰 전환, 상단 필터, 왼쪽 공사 목록 사이드바, 오른쪽 상세 작업 영역을 기준으로 한다.
- 기본 화면은 공사뷰이다.
- 공사뷰 상단 필터는 기간, 상태, 검색어, 조회, `Clear`만 제공한다.
- 공사뷰 요약은 견적 발행, 작업중, 준공 3가지로 집계한다.
- 공사 목록은 관리번호, 발주처, 선박/장비, 상태 헤더와 목록 행 구조를 사용한다.
- 상세 하위 탭은 개요, 견적서, 체크리스트, 진행 상황, 레포트, 외부 의뢰, 준공 문서 순서로 구성한다.
- 문서뷰는 견적서, 서비스 레포트, 계측 및 시험 성적서, 점검 레포트, 소견서, 수리완공사양서, 외부 의뢰 자료를 문서 중심으로 조회한다.
- 상태 흐름은 견적 상태, 수주 확정, 공사 준비, 진행 중, 레포트 작성 중, 외부 의뢰 대기, 준공 대기, 준공, 보류, 취소를 기준으로 한다.
- 체크리스트와 진행 로그는 내부 스크롤 영역으로 처리하고 전체 앱 스크롤을 만들지 않는다.
- 준공 문서가 보관 완료되면 공사 상태를 준공으로 표시할 수 있게 한다.

### 6.4 자산관리

- 고객 선박, 운용 장비, 주요 설비 기준 정보를 자산 단위로 조회한다.
- 고객관리의 선박/장비 데이터와 연결되며, 주문이나 공사에 연결된 자산은 삭제 제한 상태를 표시한다.
- 기본 목록 컬럼은 자산명, 고객, 구분, 장비 수, 최근 수정 기준으로 한다.

### 6.5 부품관리

- 요약 카드: 재고 경고, 긴급 발주 필요, 입고 대기, 출고 요청
- 최근 목록: 품목명, 현재고, 최소재고, 상태, 위치
- 빠른 작업: 재고 조회, 발주 요청, 입출고 이력

### 6.6 청구관리

- 공사, 판매, 납품 건의 청구와 수금 상태를 관리한다.
- 화면 구조는 상단 필터, 왼쪽 청구 목록, 오른쪽 청구 상세로 구성한다.
- 상세 하위 탭은 청구 정보, 수금 내역, 문서, 메모를 기준으로 한다.
- 수금 내역은 부분 수금과 수금 완료를 자동 계산하며, 예정일 경과 미수 건은 연체로 표시한다.
- 문서 추가 UI는 주문관리 문서 추가와 같은 compact 구조를 사용하고 문서ID는 자동 생성한다.

### 6.7 직원관리

- 관리부 직원에게만 노출
- 또는 `STAFF_VIEW` 권한이 있는 사용자에게 노출
- 운영용 직원 정보와 배정 현황만 보여준다
- 포함 항목: 이름, 부서, 직무, 담당 건수, 현재 상태, 연락처
- 제외 항목: 계정 잠금 해제, MFA 초기화, 권한 변경, 관리자 보안 설정

### 6.8 설정

- 이 장치 설정: 자동 로그인 사용 여부
- 서버 동기화 개인 설정: 로그인 후 기본 탭, 화면 밀도, 최근 로그인 아이디 표시 여부, 로그인 화면 기본 접속 범위 테스트값
- 현재 앱 버전
- 저장된 최근 아이디
- 서버 동기화 상태
- 업데이트 동작 안내
- 서버 동기화 개인 설정은 `identity.user_preferences` 기준으로 Electron 클라이언트와 Mac mini 로컬 관리자 화면에서 같은 값을 사용한다.
- 자동 로그인은 장치 로컬 세션 저장 여부이므로 서버 개인 설정과 분리한다.

표 스타일 기준:

- 헤더는 옅은 회색 바탕
- 본문은 흰색 배경과 얇은 행 구분선
- 상태는 배지로 표현하고 수치는 정렬을 명확히 유지

## 7. 정보 밀도 원칙

- ERP 특성상 한 화면에서 상태와 수치를 빠르게 읽을 수 있어야 한다.
- 카드 수는 최소화하고, 실제 작업은 표와 리스트 위주로 구성한다.
- 큰 여백보다 명확한 구획과 정렬을 우선한다.
- 로그인 화면보다 정보 밀도는 높이되 시각 톤은 같은 계열로 유지한다.

## 8. 공통 컴포넌트 사용 규칙

- 레이아웃: `TopBar`, `TabBar`, `SectionHeader`
- 데이터: `InfoCard`, `StatusBadge`, `DataTable`, `EmptyState`, `SettingsCard`
- 행동: `PrimaryButton`, `SecondaryButton`, `GhostButton`
- 상호작용: `ContextMenu`, `InlineEdit`, `ConfirmAction`

## 8.1 디자인 모드

- `디자인 모드`는 운영 사용자가 켜고 끄는 UI 기능이 아니라 개발자가 특정 하위 탭에 임시 적용하는 디자인 검토용 모드이다.
- 앱 화면 안에 디자인 모드 토글 버튼을 만들지 않는다.
- 디자인 모드는 요청받은 하위 탭 내부에서만 작동해야 하며, 상위 탭바, 대시보드 쉘, 다른 업무 탭에는 영향을 주지 않는다.
- 디자인 모드 대상 요소는 `data-design-id`로 식별하고, 위치와 크기 값은 탭별 localStorage key로 분리 저장한다.
- 디자인 확정 후에는 플래그를 끄거나 관련 코드를 제거할 수 있어야 한다.
- 현재 적용 기준은 주문관리 내부 workspace이며, 다른 탭에 적용할 때도 같은 스코프 제한을 유지한다.

## 9. 초기 시연 범위

- 8개 상위 탭 구조 노출
- `직원관리`의 부서 기반 조건부 노출
- 고객관리의 목록/상세/신규/우클릭 수정/선박 및 장비 관리
- 주문관리의 주문등록 하위 탭, 기간 조회, 주문 목록, 주문 내역/공사 상세
- 공사관리의 공사뷰/문서뷰, 견적서, 체크리스트, 진행 상황, 레포트, 외부 의뢰, 준공 문서
- 자산관리의 선박/장비 기준 정보 조회
- 부품관리의 재고 상태 조회
- 청구관리의 청구 목록, 수금 내역, 문서, 메모 프론트 시연 구조
- `설정` 탭의 로컬 설정과 서버 동기화 개인 설정 저장
- 로그아웃 버튼 동작 위치 정의

## 10. Electron 전용 작업 기준

- 일반 사용자 대시보드 수정은 `client/renderer/*` 기준으로만 진행한다.
- Windows/macOS Electron 클라이언트는 동일 renderer 코드를 공유하므로, 한 번의 UI 수정이 두 플랫폼에 함께 반영된다.
- 관리자 로컬 웹 대시보드와 일반 사용자 Electron 대시보드를 혼동하지 않는다.
- 전체 창 스크롤 대신 내부 패널 스크롤 규칙을 유지한다.
